# 2026-04-20 上线版本说明

Updated: 2026-04-20

## 1. 版本标识

- 仓库：`linxiangpk1991/genergi`
- 分支：`codex/unify-media-model-slots`
- 发布方式：`pnpm deploy:production`
- 线上入口：[https://ai.genergius.com](https://ai.genergius.com)

这次上线的主题不是新增一个“新功能入口”，而是把模型控制面、任务启动页、worker 运行链和交接文档统一收敛到当前真实执行链，消除旧六槽位心智造成的误导。

---

## 2. 本次上线范围

### 2.1 媒体模型槽位正式收敛为四槽位

本次已统一为以下运行时槽位：

- `textModel`
- `imageModel`
- `videoModel`
- `ttsProvider`

已移除旧的误导性槽位心智：

- `imageDraftModel`
- `imageFinalModel`
- `videoDraftModel`
- `videoFinalModel`

这意味着：

- 当前任务启动即直接进入真实生成链
- 系统不再对外暗示“草图 / 终图、草稿 / 终稿视频”的双阶段执行
- 默认值、任务级覆盖、冻结快照与 worker 消费现在都指向同一套真实槽位

### 2.2 控制面 / 运行时 / UI 一次性对齐

本次一起更新了：

- `packages/shared`
  - 槽位 schema
  - defaults / override / snapshot 结构
  - legacy defaults 折叠迁移逻辑
- `packages/config`
  - seed 默认值改为四槽位
- `apps/api`
  - registry / defaults / resolver / task-store 全部切到四槽位
- `apps/worker`
  - 只消费统一后的 `imageModel` / `videoModel`
- `apps/web`
  - 任务级高级覆盖只展示四槽位
  - Keyframe Review / Help Center / Release Timeline 同步改文案

### 2.3 旧任务清理与 rollout 验证链路

这次还补了两条上线配套能力：

- `pnpm cleanup:legacy-tasks`
  - 清空已确认允许删除的旧任务摘要 / 详情 / 资产 / 导出目录
- `pnpm validate:unified-media-slots`
  - 自动清理旧任务数据
  - 创建一条 smoke task
  - 输出 frozen snapshot 的四槽位证据

这让本次改造从“代码改完”提升为“上线过程可验证”。

## 3. 代码层改动摘要

### 3.1 Shared / Persistence

关键变化包括：

- `modelSlotTypeSchema` 收敛到四槽位
- `TaskRunConfig` 收敛到：
  - `textModel`
  - `imageModel`
  - `videoModel`
  - `ttsProvider`
- `readModelDefaults()` 支持把 legacy 六槽位 defaults 折叠到四槽位：
  - 图片优先取旧 `imageFinalModel`
  - 视频优先取旧 `videoFinalModel`

### 3.2 API / Resolver / Task Freeze

当前后端只会解析并冻结：

- `textModel`
- `imageModel`
- `videoModel`
- `ttsProvider`

任务创建后写入 `taskRunConfig.slotSnapshots` 的内容已经与真实执行链完全一致。

### 3.3 Worker Runtime

worker 不再依赖旧的 draft/final 媒体槽位语义。

现在：

- 图像生成只读 `imageModel`
- 视频生成只读 `videoModel`
- TTS 继续按 `ttsProvider`

### 3.4 Web / Docs / Release Notes

运营侧可见内容已同步更新：

- 模型控制中心
- Defaults Center
- 任务页高级覆盖
- Keyframe Review 指标卡
- Help Center 功能说明 / 工作流说明 / 时间线
- handover 文档

旧六槽位只保留在：

- legacy defaults 迁移代码
- 历史 spec / plan 的说明上下文

并且旧 `2026-04-19` 六槽位控制面设计文档已经明确标注为被新四槽位设计取代。

## 5. 对团队的意义

这次上线后的直接收益是：

1. 运营不会再被伪阶段槽位误导
2. 新任务的默认解析、任务覆盖和 worker 消费终于讲的是同一种语言
3. 后续如果引入真正的多阶段视觉链路，可以在新 pipeline 里显式设计，而不是背着旧命名债务继续演进
4. 上线与回归有了明确的专项 smoke 入口，接手成本更低
