# GENERGI 项目完整说明

Updated: 2026-04-17

## 1. 项目概览

### 1.1 项目名称

- 主品牌：`GENERGI`
- 产品展示名：`GENERGI 自动化视频平台`
- 公司 / 域名层：`Genergius`
- 主入口域名：`ai.genergius.com`

### 1.2 项目定位

`GENERGI` 是一个面向中文运营团队、服务于海外英语市场的短视频自动化生产平台。

它不是通用型剪辑器，也不是桌面内容工具，而是一个 web-first 的“视频生产中台”：

- 中文后台操作
- 英语内容输出
- 面向 TikTok / Instagram Reels / YouTube Shorts
- 追求批量化、可审阅、可控成本、可持续部署

### 1.3 当前阶段

当前仓库处于 `Phase 1` 基础建设阶段，已经具备一套可运行的 web / api / worker 基线，并已连通真实媒体链中的关键节点。

但它还不是“所有能力都完全打磨好的正式版平台”。当前更准确的状态是：

- 核心链路已通
- 关键页面已落地
- 部署模型已稳定
- 文档与团队协作基础已建立
- 仍有部分模块属于 Phase 1 占位实现或待深化实现

## 2. 项目目标

### 2.1 业务目标

项目希望解决的问题是：

- 让运营团队可以快速发起英语短视频生产任务
- 让内容从脚本、分镜、关键帧到成片有结构化流程
- 让模型与 provider 可替换，避免把系统绑定到单一供应商
- 让部署、回滚、热补丁具备长期可维护性

### 2.2 产品目标

当前产品目标可以概括为四条：

1. 建立可运行的短视频自动生产基础平台
2. 建立明确的审阅关卡
3. 建立素材资产沉淀与导出入口
4. 建立适合团队协作的工程与部署基线

## 4. 当前技术架构

### 4.1 Monorepo 结构

仓库根目录：`E:\genergi`

核心结构如下：

- `apps/web`：React + Vite 控制台前端
- `apps/api`：Hono API 层
- `apps/worker`：Node.js + BullMQ 执行引擎
- `packages/shared`：共享 schema、任务 / 用户 / 资产持久化
- `packages/config`：品牌、渠道、模式与模型配置
- `packages/ui`：GENERGI 设计系统组件导出入口
- `docs/architecture`：架构、部署、UI 基线与约束文档
- `docs/handover`：交接文档与团队接手资料

### 4.2 运行形态

系统由三个逻辑运行单元组成：

1. `web`
   - 负责中文操作台
   - 发起任务、查看任务、执行审阅、查看资产、管理用户

2. `api`
   - 负责鉴权、任务创建、任务详情、资产下载与预览、运行时状态、用户中心接口

3. `worker`
   - 负责异步执行任务
   - 处理脚本增强、TTS、视频生成、关键帧生成、成片拼接、资产登记

### 4.3 当前持久化方式

当前 Phase 1 默认使用文件持久化，而不是数据库主导。

默认数据目录：

- `.data/`

可通过环境变量覆盖：

- `GENERGI_DATA_DIR`

当前持久化的对象包括：

- 任务摘要
- 任务详情
- 运行时状态
- 资产清单
- 用户列表

这套设计的优点是启动快、迁移轻；缺点是更适合早期阶段，不适合长期高并发、多实例共享场景。

## 6. 任务与内容生产流程

### 6.1 当前端到端流程

从代码视角看，一个任务的当前链路是：

1. 前端在任务启动工作台提交任务
2. API 记录目标正片总时长，并创建任务摘要与任务详情
3. API 将任务加入 `BullMQ` 队列
4. Worker 消费任务
5. Worker 尝试用文本 provider 改写脚本
6. Worker 基于“脚本 + 正片总时长”重新规划真实分镜
7. Worker 生成本地脚本文件与 storyboard JSON
8. Worker 用 `Edge TTS` 生成音频与字幕
9. Worker 按每个 scene 的时长逐段生成场景视频
10. Worker 生成关键帧包
11. 如果关键帧生成失败，则回退为从视频抽帧
12. Worker 将多段场景视频拼接成完整画面轨
13. Worker 将 narration 音频 mux 到最终画面轨
14. Worker 把所有结果登记为资产
15. 前端资产中心可查看与下载

### 6.2 当前真实实现范围

这里需要特别诚实说明当前代码的真实状态：

- 任务详情会根据正片总时长自动生成多 scene 分镜结构
- 当前支持的正片总时长档位为 `15 / 30 / 45 / 60` 秒
- 不同时长会映射到不同 scene 数与每段时长
- 关键帧包会按所有 scene 循环生成
- 视频生成会按所有 scenes 逐段执行
- 最终成片是“多段场景视频拼接 + narration 音频”的合成结果

这意味着：

- 系统已经具备“可定制正片总时长 -> 自动分镜 -> 多段视频 -> 最终成片”的基础主链
- 但严格意义上的高级节奏控制、scene 级手动时长编辑、精细镜头编排仍然不是最终成熟形态

这是当前交接时必须明确给团队的事实。

## 7. 模式、模型与 Provider 策略

这是本项目最核心的部分之一。

### 7.1 设计原则

系统没有把文本、图像、视频、TTS 绑死在同一家 provider 上，而是采用“模式注册 + provider 可替换”策略：

- 模式决定默认模型组合与预算
- 运行时可通过环境变量替换 provider 接入
- Worker 主逻辑尽量不随 provider 切换而重写

### 7.2 当前创作模式

系统目前有两种生产模式：

#### 1. `mass_production`

定位：

- 量产模式
- 更强调效率、批量生产、成本控制

默认配置：

- 文本模型：`Claude Opus 4.6`
- 草图图像模型：`Gemini 3.1 Flash Image Preview`
- 正式图像模型：`Gemini 3 Pro Image Preview`
- 草稿视频模型：`Veo 3.1 Fast`
- 正式视频模型：`Veo 3.1 Portrait`
- TTS：`edge-tts`
- 预算上限：`¥3`
- 分镜审阅：默认关闭
- 关键帧审阅：默认开启

#### 2. `high_quality`

定位：

- 高质量模式
- 更强调品牌表现、审阅质量、画面效果

默认配置：

- 文本模型：`Claude Opus 4.6`
- 草图图像模型：`Gemini 3 Pro Image Preview`
- 正式图像模型：`Gemini 3 Pro Image Preview 2k`
- 草稿视频模型：`Veo 3.1 Portrait`
- 正式视频模型：`Veo 3.1 Portrait HD`
- TTS：`edge-tts`
- 预算上限：`¥5`
- 分镜审阅：默认开启
- 关键帧审阅：默认开启

### 7.3 当前模型注册表

从 `packages/config/src/index.ts` 看，系统把模式和模型绑定为默认组合。

这意味着当前“模型注册中心”的能力已经具备：

- 模式到模型组合的映射
- 模式到预算的映射
- 模式到审阅门控策略的映射

### 7.4 文本 Provider

当前文本侧支持的协议：

- `anthropic-native`
- `openai-compatible`

文本改写逻辑用于：

- 把输入脚本改得更“native English”
- 更贴近短视频平台文案风格

运行条件：

- `GENERGI_TEXT_PROVIDER`
- `GENERGI_TEXT_BASE_URL`
- `GENERGI_TEXT_API_KEY`
- `GENERGI_TEXT_MODEL`

如果这些变量未配置：

- worker 会跳过改写
- 直接使用原始脚本继续流程

### 7.5 图像 Provider

当前图像生成走媒体网关：

- `GENERGI_MEDIA_GATEWAY_BASE_URL`
- `GENERGI_MEDIA_GATEWAY_API_KEY`

图像模型 alias 到真实模型名的映射如下：

- `image.draft` -> `gemini-3.1-flash-image-preview`
- `image.final` -> `gemini-3-pro-image-preview`
- `image.premium` -> `gemini-3-pro-image-preview-2k`

系统还做了一个保守兼容策略：

- 如果模型名里带 `2k / 4k / portrait / landscape` 等字样，但无法稳定映射，会回退到较稳定的 `gemini-3.1-flash-image-preview`

### 7.6 视频 Provider

当前视频生成同样走媒体网关。

关键环境变量：

- `GENERGI_MEDIA_GATEWAY_BASE_URL`
- `GENERGI_MEDIA_GATEWAY_API_KEY`
- `GENERGI_VIDEO_MODEL`

如果不显式指定视频模型：

- worker 会优先用任务配置里的 `videoDraftModel.id`

### 7.7 TTS Provider

Phase 1 默认 TTS 方案：

- `Edge TTS`

原因：

- 低成本
- 快速可用
- 不会在早期阶段因为预算卡住整条链路

关键环境变量：

- `GENERGI_TTS_PROVIDER=edge-tts`
- `GENERGI_TTS_VOICE=en-US-AvaMultilingualNeural`

### 7.8 Fallback 逻辑

当前最重要的 fallback 设计是：

- 如果关键帧图片生成失败或超时
- 自动退回为“从真实视频抽帧生成关键帧包”

这让系统即使在图片侧不稳定时，也尽量保证有可交付结果。

## 13. 部署与运维

### 13.1 生产部署模型

正式环境采用稳定部署模型，而不是把 Docker Compose 当主平台：

- `/opt/genergi/releases/<timestamp>`
- `/opt/genergi/current`
- `/opt/genergi/current.prev`
- `/opt/genergi/shared/`

### 13.2 运行职责

- `nginx`：静态站点 + `/api` 反向代理
- `systemd`：守护 `api` / `worker`

### 13.3 发布方式

当前提供两条发布路径：

1. `pnpm deploy:production`
   - 标准 release archive 发布

2. `pnpm deploy:hotfix`
   - 变更文件热补丁同步
   - 适合 SSH / SCP 抖动时快速止血

### 13.4 运维原则

- 不把运行期数据写进 release 目录
- 不因网络抖动死磕大包上传
- 发布必须可回滚
- 验收必须包含健康检查与真实业务 smoke

## 15. 当前实现边界与已知限制

为了避免团队误判，下面这些点必须明确：

### 15.1 已经打通的

- 中文后台基本壳层
- 任务创建
- 正片总时长档位配置
- 任务详情与 scene 结构
- 用户中心
- 资产中心
- 文本改写接入
- TTS
- 多段视频生成
- 多段视频拼接
- 关键帧生成
- 图片失败时视频抽帧 fallback
- 最终成片输出
- 稳定部署脚本

### 15.2 仍属 Phase 1 或半实装状态的

- 分镜 / 关键帧审批结果的完整状态闭环
- 高级参数与后端执行配置的完全打通
- 更细的时长控制（例如 scene 级手动时长、节奏模板、平台专属节奏）
- 更细粒度的预算模型
- 多角色权限系统
- 对象存储 / 版本化资产中心
- 更完整的数据库化持久化

### 15.3 当前最应避免的误解

不要把当前系统理解成：

- 已经完整商用化的多镜头视频编排平台
- 已经完成所有审批闭环的中后台系统
- 已经完成数据库化、队列化、对象存储化的终局架构

更准确的理解应是：

- 一个真实可运行、主链打通、具备继续迭代基础的 web-first 视频生产平台
