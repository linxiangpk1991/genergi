# GENERGI 模型控制面使用说明

Updated: 2026-04-19

## 1. 这套系统解决什么问题

模型控制面的目标是把“改模型”从代码层抽出来，变成系统内可管理、可验证、可冻结的一套正式能力。

当前已经支持：

- 后台新增 Provider
- 后台新增 Model
- Provider / Model 校验
- 全局默认值
- 模式默认值
- 任务页高级覆盖
- 任务创建时冻结模型快照

它解决的是三个痛点：

1. 运营不需要再改代码才能换模型
2. 工程可以保证只有通过校验的记录才进入可选池
3. 任务一旦创建，运行链路不会被后续默认值变更污染

---

## 2. 三层优先级

当前固定优先级：

1. 任务覆盖
2. 模式默认
3. 全局默认

解释：

- 全局默认是兜底
- 模式默认是“量产 / 高质量”这两条生产线的常规配置
- 任务覆盖只影响当前任务一次

创建任务时，系统会把最终解析结果冻结进 `taskRunConfig.slotSnapshots`。

这意味着：

- 改默认值不会回写旧任务
- 改 provider / model 也不会让历史任务半路换发动机

---

## 3. 页面入口

后台入口：

- `模型控制中心`

下设三页：

1. `Provider 管理`
2. `Model Registry`
3. `Defaults Center`

任务页入口：

- `任务启动` 页中的 `任务级高级覆盖`

---

## 4. Provider 管理怎么用

### 4.1 新增 Provider

必填字段：

- `providerKey`
- `displayName`
- `providerType`
- `authType`

按需要填写：

- `endpointUrl`
- `secret`

说明：

- 明文 secret 不会回显给前端
- 系统只会显示掩码状态
- endpoint / secret 会在服务端加密保存

### 4.2 状态说明

- `draft`
  - 新建完成，但还没验证
- `validating`
  - 正在校验
- `available`
  - 可被下游 model 使用
- `invalid`
  - 校验失败，需要修配置
- `disabled`
  - 手动停用，不进入可选池
- `deprecated`
  - 历史保留，不建议继续使用

### 4.3 校验规则

当前版本只做接入级校验，不做轻量生成 smoke。

会检查：

- endpoint 格式
- auth 是否完整
- 必要 secret 是否存在

通过后：

- Provider 自动进入 `available`

---

## 5. Model Registry 怎么用

### 5.1 一个 Model 记录包含什么

- `modelKey`
- `providerId`
- `slotType`
- `providerModelId`
- `displayName`
- `capabilityJson`

### 5.2 六个槽位

当前精细管理这六个槽位：

- `textModel`
- `imageDraftModel`
- `imageFinalModel`
- `videoDraftModel`
- `videoFinalModel`
- `ttsProvider`

说明：

- 文本 / 图像 / 视频槽位按 model 记录管理
- `ttsProvider` 当前按 provider 解析，但仍进入同一套 defaults / override / snapshot 机制

### 5.3 校验规则

模型校验会检查：

- 绑定的 provider 是否 `available`
- `providerModelId` 是否已填写
- 槽位与能力元数据是否匹配

通过后：

- Model 自动进入 `available`
- 自动进入任务页高级覆盖可选池

---

## 6. Defaults Center 怎么用

### 6.1 全局默认

作用：

- 给所有模式提供兜底值

适合放：

- 最稳的通用默认模型
- 不希望频繁变动的基础 provider

### 6.2 模式默认

当前两个模式：

- `mass_production`
- `high_quality`

作用：

- 按业务目标覆盖全局默认

建议：

- 量产模式优先效率和成本
- 高质量模式优先画面和审阅质量

### 6.3 提交规则

Defaults Center 保存的是**真实后端默认值**，不是前端草稿模拟。

保存后影响：

- 新任务的默认解析
- 任务页高级覆盖摘要

不会影响：

- 已经创建完成的历史任务

---

## 7. 任务页高级覆盖怎么用

任务页中的 `任务级高级覆盖` 适合处理：

- 这次只想换文案模型
- 这次只想切视频终稿模型
- 这次想临时换一个 TTS provider

规则：

- 只能从 `available` 池里选
- 不支持手输临时 ID
- 只提交显式改动项
- 留空的槽位继续走模式默认 / 全局默认

使用建议：

- 没特殊需要时不要开高级覆盖
- 只改你明确要替换的槽位
- 如果是长期策略，去 Defaults Center 改，不要每次都在任务页覆盖

---

## 8. 数据如何落盘

当前文件持久化对象包括：

- `providers.json`
- `models.json`
- `model-defaults.json`

运行时任务快照落在：

- `tasks.json`
- `task-details.json`

其中任务详情中的 `taskRunConfig.slotSnapshots` 是最关键的冻结数据。

---

## 9. 工程侧必须知道的规则

1. `packages/config` 现在只是 seed 数据，不再是运行时唯一真相
2. Worker 必须优先消费冻结后的 `slotSnapshots`
3. 不允许把 `draft / invalid / disabled / deprecated` 记录放进可选池
4. 不允许把原始 secret 返回给前端
5. 任务创建时必须冻结快照，不能只存 alias

---

## 10. 常见排查顺序

### 症状：任务页看不到某个模型

先查：

1. Provider 是否 `available`
2. Model 是否 `available`
3. 槽位是否匹配
4. 当前模式是否真的读到了新的 defaults

### 症状：改了默认值但新任务没生效

先查：

1. 改的是全局默认还是模式默认
2. 当前任务模式是什么
3. 任务页是否开了高级覆盖
4. 后端 `model-defaults.json` 是否真的落盘

### 症状：任务运行时模型不对

先查：

1. 任务详情中的 `taskRunConfig.slotSnapshots`
2. 创建任务时是否已经被冻结旧值
3. Worker 是否按冻结快照消费，而不是回退到静态配置

---

## 11. 当前已知边界

当前版本已经够用，但还不是终态。

还可以继续增强的方向包括：

- Provider / Model 操作审计日志
- 灰度发布与回滚策略
- 更细的可观测性
- 更高级的 fallback / cost / latency 面板
- 更强的模型健康检查

但这些属于下一阶段增强，不影响当前控制面作为正式运行时控制入口使用。
