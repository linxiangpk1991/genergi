# GENERGI 关键画面优先审核流视频生产设计

Date: 2026-04-20
Status: 已与运营方向对齐，待进入实现规划
Scope: 为 GENERGI 设计一套以“关键画面优先”为核心的视频生产链，支持模式级全自动 / 审核流切换、任务内审核记录、项目级审核库沉淀，以及终端尺寸预设对整条生成链的前置约束。

## 1. 背景与问题

当前系统已经具备一条可运行的视频生成链：

1. 用户提交内容母本
2. API 创建任务并冻结运行配置
3. Worker 尝试改写脚本
4. Worker 生成旁白与字幕
5. Worker 逐段生成视频
6. Worker 生成关键画面包
7. Worker 拼接视频并叠加旁白

这条链已经能产出真实结果，但存在四个关键问题：

### 1.1 关键画面没有进入视频主链

当前关键画面是在视频生成之后才生成的，图片模型产物不参与视频生成输入。

这导致：

- 图片模型对最终成片质量影响偏弱
- 关键画面更像审阅参考图，而不是视频生成锚点
- 视频模型仍然主要按文字重新理解画面

### 1.2 文本模型输出不够“前置冻结”

当前文本模型虽然会尝试改写脚本和重写分镜，但系统并没有把“整条视频的执行蓝图”显式定义为第一阶段正式产物。

这导致：

- 图片提示词与视频提示词可能被视为下游临时结果
- 后续审核对象不够清晰
- 返工时缺少统一的版本主键

### 1.3 审核流还不是昂贵阶段的硬门禁

当前系统已有真实 review 状态与 review 写回，但并没有做到：

- 先完成关键画面与提示词审核
- 审核通过后才开始完整视频生成

这会让昂贵阶段过早启动。

### 1.4 尺寸约束进入链路过晚

当前系统有长宽比字段，但“手机 / 平板”等终端规格还没有作为前置创作约束进入文本规划、图片提示词和视频提示词。

这会导致：

- 画面构图不稳定
- 终端适配更像事后裁切
- 审核时缺少明确终端上下文

## 2. 核心决策

本次设计明确做出以下决策：

### 2.1 视频生成主链改为关键画面优先

系统的推荐主链由当前的“视频优先”调整为：

`内容母本 -> 执行蓝图 -> 关键画面 -> 关键画面审核/确认 -> 视频生成 -> 旁白/字幕 -> 最终合成`

### 2.2 文本模型在开头一次性输出整套执行蓝图

文本模型不再只承担“旁白改写”角色，而是要在第一阶段一次性输出：

- 全局风格约束
- 全局主体 / 产品 / 场景约束
- 总旁白稿
- 每个分镜的分镜契约
- 每个分镜的图片提示词
- 每个分镜的视频提示词
- 尺寸约束下的构图要求

### 2.3 模式级决定全自动 / 审核流

不在任务创建时额外增加运行方式选择，而是在模式中直接定义：

- `全自动`
- `审核流`

任务只继承模式，不额外覆盖。

### 2.4 审核流采用整任务放行

第一版不做“分镜逐段放行”，而采用：

- 整条任务的关键画面、提示词、旁白稿统一进入审核库
- 整任务审核通过
- 再进入完整分镜视频生成

底层数据可保留分镜级审核状态，但执行策略先按整任务放行实现。

### 2.5 审核结果同时沉淀到两层

审核流产物需要同时进入：

- `任务内审核记录`
- `项目级审核库`

其中：

- 任务内审核记录保留完整版本链
- 项目级审核库只沉淀审核通过版本

### 2.6 尺寸是前置创作约束，不是最终导出参数

尺寸预设必须在任务创建时冻结，并前置影响：

- 文本模型输出
- 图片提示词
- 视频提示词
- 图片生成
- 视频生成
- 审核页展示

## 3. 核心对象与层级关系

## 3.1 项目

项目是任务的上级容器，对应同一客户、产品线、内容系列或长期生产主题。

项目至少保存：

- 项目名称
- 品牌 / 产品方向
- 默认渠道集合
- 可复用风格约束
- 项目级审核库引用

项目不直接决定一次生成的具体内容，但负责承载长期沉淀与复用。

## 3.2 模式

模式是生产策略包，不是内容容器。

模式至少保存：

- 执行方式：全自动 / 审核流
- 默认文本模型
- 默认图片模型
- 默认视频模型
- 默认配音提供方
- 默认预算级别
- 默认渠道尺寸策略

模式决定“怎么生成”，不决定“生成什么”。

## 3.3 任务

任务是一次具体的视频生产请求。

任务创建时必须冻结：

- 所属项目
- 所属模式
- 所属渠道
- 最终终端预设
- 最终尺寸规格
- 当次内容母本
- 冻结后的运行配置

任务是后续状态流转、审核、资产、日志和重试的主键对象。

## 3.4 执行蓝图

执行蓝图是本次设计里最关键的新对象。

它是任务在真正进入媒体生成前的正式拍摄计划书，必须成为后续所有模型的统一输入来源。

执行蓝图至少包含：

- 全局风格约束
- 全局角色 / 产品 / 场景约束
- 总旁白稿
- 当前尺寸规格
- 当前审核策略
- 全部分镜契约

后续图片模型、视频模型、配音和字幕都消费执行蓝图，而不是直接消费原始内容母本。

## 3.5 分镜契约

分镜契约是执行蓝图中的最小执行单元。

每个分镜契约至少包含：

- 分镜编号
- 分镜目标
- 分镜旁白
- 起始关键画面描述
- 图片提示词
- 视频提示词
- 起始状态
- 结束状态
- 时长
- 转场方式
- 不可漂移约束

图片模型与视频模型的输入都应来自这里。

## 3.6 任务内审核记录

任务内审核记录只服务当前任务。

它需要保存：

- 执行蓝图版本
- 每个分镜的关键画面
- 图片提示词
- 视频提示词
- 旁白稿
- 审核意见
- 审核状态
- 驳回后的重生成版本链

它的重点是可追溯和可比较。

## 3.7 项目级审核库

项目级审核库只沉淀审核通过版本。

它保存：

- 审核通过的执行蓝图
- 审核通过的分镜契约
- 审核通过的关键画面
- 审核通过的提示词组合
- 审核通过的旁白表达模板

它的目标是支持下一条任务复用，而不是替代任务级历史。

## 4. 尺寸预设设计

## 4.1 渠道默认值

渠道只给默认值，不给最终值。

第一版默认映射为：

- TikTok -> 手机竖屏
- Instagram Reels -> 手机竖屏
- YouTube Shorts -> 手机竖屏

## 4.2 任务级终端预设

任务创建时允许从以下四个终端预设中选择最终目标：

1. 手机竖屏：`1080 x 1920`
2. 手机横屏：`1920 x 1080`
3. 平板竖屏：`1536 x 2048`
4. 平板横屏：`2048 x 1536`

## 4.3 尺寸规格对象

任务创建时必须冻结为一份“尺寸规格对象”，而不是只存一个比例字符串。

建议最少包含：

- `terminalPresetId`
- `width`
- `height`
- `aspectRatio`
- `safeArea`
- `compositionGuideline`
- `motionGuideline`

## 4.4 尺寸如何约束文本模型

文本模型必须在第一阶段就接收尺寸规格对象。

它会影响：

- 分镜数量和节奏
- 构图方向
- 主体分布方式
- 图片提示词中的构图限制
- 视频提示词中的镜头运动限制

因此尺寸必须在任务创建时就进入执行蓝图，而不是等渲染时再处理。

## 5. 执行蓝图输出要求

## 5.1 全局层

文本模型第一阶段必须输出整条视频的全局蓝图，至少包括：

- 全局主题
- 全局风格方向
- 全局主体设定
- 全局产品设定
- 全局背景约束
- 全局负面约束
- 总旁白稿

## 5.2 分镜层

对每个分镜，文本模型必须一次性输出：

- 分镜目标
- 分镜旁白
- 起始关键画面描述
- 图片提示词
- 视频提示词
- 起始状态
- 结束状态
- 时长
- 转场方式

## 5.3 图片提示词与视频提示词的关系

视频提示词不能只由图片提示词推导。

正确关系应为：

- 母本是唯一内容源头
- 分镜契约是统一执行源头
- 图片提示词负责静态画面
- 视频提示词负责动作、镜头、节奏与结尾状态

也就是说，两者都应来自分镜契约，而不是彼此串推。

## 6. 全自动工作流

全自动模式的目标是：不等待人工审核，但仍保留完整蓝图和关键画面优先逻辑。

建议流程：

1. 创建任务
2. 冻结运行配置与尺寸规格
3. 生成执行蓝图
4. 生成全部关键画面
5. 直接进入分镜视频生成
6. 生成旁白音频
7. 生成字幕
8. 拼接分镜视频
9. 合成最终成片
10. 登记资产

全自动模式仍应保留任务内执行记录，但不进入人工等待审核状态。

## 7. 审核流工作流

审核流的目标是：在最昂贵的视频生成之前，先把整条任务的蓝图与关键画面固定下来。

建议流程：

1. 创建任务
2. 冻结运行配置与尺寸规格
3. 生成执行蓝图
4. 生成全部关键画面
5. 将以下内容写入任务内审核记录：
   - 执行蓝图版本
   - 总旁白稿
   - 每个分镜契约
   - 每个分镜的图片提示词
   - 每个分镜的视频提示词
   - 每个分镜关键画面
   - 当前尺寸规格
6. 任务进入等待审核状态
7. 审核通过后，同步通过版本到项目级审核库
8. 再继续执行：
   - 分镜视频生成
   - 旁白生成
   - 字幕生成
   - 最终合成

## 7.1 硬门禁规则

审核流必须满足：

**在关键画面与提示词审核通过之前，不允许开始完整分镜视频生成。**

这条规则是审核流的核心，不得退化成“边审边先生成视频”。

## 7.2 驳回规则

第一版采用整任务驳回与重生成为主，不做分镜逐段放行。

当审核驳回时：

- 当前版本执行蓝图标记为驳回
- 不进入视频生成
- 返回蓝图修订阶段
- 重新生成新的蓝图版本和关键画面
- 再重新进入整任务审核

第一版的返工单位是“执行蓝图版本”，不是“单个视频片段”。

## 8. 视频生成方式设计

## 8.1 目标主链

视频生成推荐主链改为：

`关键画面 + 视频提示词 -> 分镜视频`

即视频模型应围绕已生成好的关键画面生成运动，而不是只根据文字重新推断整个场景。

## 8.2 兜底链

为了保证稳定性，应保留一条兜底链：

`仅视频提示词 -> 分镜视频`

当图片驱动视频生成不可用、超时或供应方能力不稳定时，任务可退化到兜底链，但必须留下清晰执行记录。

## 8.3 当前系统与目标系统的区别

当前真实系统更接近：

`文字提示 -> 分镜视频 -> 关键画面`

目标系统应改为：

`分镜契约 -> 关键画面 -> 关键画面 + 视频提示词 -> 分镜视频`

## 9. 旁白与字幕策略

## 9.1 当前定位

当前平台的视频内容仍以统一 narrator 旁白为主，而不是依赖视频模型自己带回的原生对白或原生语音。

因此：

- 旁白层仍然独立存在
- 字幕层仍然独立存在

## 9.2 产物契约

无论是全自动还是审核流，最终都应继续保证：

- 独立音频资产
- 独立字幕资产
- 最终成片资产

也就是继续维持：

- `narration audio`
- `subtitles`
- `final video`

后续即使支持视频原生音频，也不应轻易推翻这一资产契约。

## 10. 数据持久化建议

## 10.1 新增 / 扩展字段

任务级建议新增：

- `projectId`
- `executionMode`
- `terminalPresetId`
- `renderSpecJson`
- `blueprintVersion`
- `blueprintStatus`

分镜级建议新增或重构：

- `sceneGoal`
- `voiceoverScript`
- `startFrameDescription`
- `imagePrompt`
- `videoPrompt`
- `startFrameIntent`
- `endFrameIntent`
- `continuityConstraints`

审核记录建议新增：

- `taskBlueprintRecords`
- `taskBlueprintReviewRecords`
- `projectApprovedBlueprintLibrary`

## 10.2 版本策略

执行蓝图必须带版本号。

原则是：

- 每次文本模型重写蓝图，产生一个新蓝图版本
- 每次图片重生成，只更新该蓝图版本下的关键画面结果
- 审核通过时锁定当前通过版本

## 11. 状态机建议

建议将任务主流程状态细化为：

- 已创建
- 生成蓝图中
- 生成关键画面中
- 等待审核
- 审核驳回
- 审核通过
- 生成分镜视频中
- 生成旁白中
- 生成字幕中
- 最终合成中
- 已完成
- 已失败

这些状态可以映射到现有基础状态机，但前端展示应明确体现新阶段含义。

## 12. 第一版不做的事

本次设计明确不在第一版实现以下内容：

- 分镜逐段放行
- 分镜逐段审核后立即生成对应视频
- 项目级复杂模板市场
- 多终端同时导出多版本
- 视频模型原生音频作为默认路径
- 复杂的自动局部修订逻辑

这些都应在第一版稳定后再评估。

## 13. 验收标准

本次设计进入实现后，必须满足以下验收条件：

1. 文本模型第一阶段能一次性生成整套执行蓝图
2. 执行蓝图中包含完整分镜契约、图片提示词和视频提示词
3. 图片模型先生成关键画面，视频模型后生成分镜视频
4. 审核流下，关键画面与提示词未审核通过前，不进入完整视频生成
5. 审核通过版本能同步到项目级审核库
6. 任务创建时可冻结终端预设与明确尺寸
7. 尺寸约束能影响文本模型、图片模型和视频模型的输入
8. 全自动链与审核流链都能跑通
9. 现有资产中心仍能统一查看音频、字幕、关键画面与成片

## 14. 建议实施顺序

建议后续实现顺序为：

1. 补齐项目、模式执行方式、终端尺寸预设的数据结构
2. 新增执行蓝图与分镜契约持久化结构
3. 改造文本模型输出，生成完整蓝图
4. 把图片生成前移到视频生成之前
5. 把视频生成改成“关键画面 + 视频提示词”驱动
6. 加入任务内审核记录与项目级审核库
7. 把审核流设成真正硬门禁
8. 最后再补前端审核页与项目视图

## 15. 推荐结论

这次迭代应被视为 GENERGI 从“可运行的视频生成系统”升级到“可审核、可沉淀、可复用的视频生产平台”的关键转折点。

最重要的原则必须固定下来：

1. 文本模型一开始就输出整套执行蓝图
2. 关键画面优先于视频生成
3. 审核流先审蓝图与关键画面，再放行视频生成
4. 尺寸是前置创作约束，不是最终导出参数
5. 审核通过内容必须沉淀到项目级审核库

如果这五点不被同时落实，系统仍会停留在“能出视频”，而不是“能稳定生产视频”的阶段。
