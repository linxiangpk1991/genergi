# GENERGI 稳定部署能力说明

## 目标

这份说明定义 `GENERGI` 的**可复用部署能力**，不是一次性的临场排障记录。

我们要解决的不是“这次怎么传上去”，而是长期会反复出现的这些问题：

- SSH 抖动
- `scp` 大包传输中断
- 远端 build 后的服务重启与回滚
- `nginx` / `systemd` 验收路径不稳定

## 双通道发布模型

### 1. 正式发布：archive release

主路径仍然是：

- 本地打包 release archive
- 上传到远端 `/tmp`
- 远端解压到 `/opt/genergi/releases/<timestamp>`
- 切换 `current` / `current.prev`
- 重启 `systemd`
- 验收

对应脚本：

- `scripts/deploy/deploy-production.mjs`

### 2. 热补丁发布：changed-files sync

当 archive 上传因为 SSH/SCP 抖动不稳定时，不再继续重复撞大包传输，而是切换为：

- 只同步本次变化的文件
- 远端在 `current` 目录内重新 build
- restart `api` / `worker`
- `nginx -t` / reload
- 验收

对应脚本：

- `scripts/deploy/deploy-hotfix-sync.mjs`

## 远端工具规则

统一通过：

- `tools/node/remote.mjs`

来处理：

- `ssh`
- `scp`
- remote script

### 已补能力

- 远端命令自动重试
- 远端脚本自动重试
- `scp` 自动重试
- 网络层错误识别

## 什么时候切换到 hotfix sync

满足任意一个条件就优先用热补丁路径：

- `scp` / archive 上传多次失败
- 只改了少量文件
- 线上已经有稳定 `current` release
- 要求快速止血而不是完整换版

## 验收基线

每次发布至少要过这几关：

1. `api/health`
2. `worker` 进程存活
3. `nginx -t`
4. 外网入口可访问
5. 至少一条真实业务 smoke

## 操作原则

- 不在 SSH 抖动时死磕同一条大包上传
- 不让发布过程只剩“碰运气”
- 不把远端失败的临时修复留在脑子里，必须写成脚本或文档
