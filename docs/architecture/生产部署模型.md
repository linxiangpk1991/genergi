# GENERGI 生产部署模型

## 目标

`GENERGI 自动化视频平台` 的生产环境不再把 `Docker Compose` 当作主平台部署模式。  
正式环境统一采用已经被现网稳定项目验证过的模型：

- 时间戳 release 目录
- `current` / `current.prev` 原子切换
- `systemd` 常驻 `api` / `worker`
- `nginx` 负责静态站点与 `/api` 反向代理

## 目录结构

```text
/opt/genergi/
  releases/<timestamp>/
  current -> releases/<timestamp>
  current.prev -> releases/<previous>
  shared/
    env/
    logs/
    assets/
    tmp/
    data/
```

## 核心原则

### 1. release 目录只放代码与构建产物

release 目录是可替换的，不允许把运行期数据绑进去。

### 2. 运行期数据必须放在稳定共享目录

例如：

- 任务数据
- 资产缓存
- 日志
- 模板

都必须放在 `/opt/genergi/shared/*` 下。

### 3. `apps/web` 走静态站点

前端是 `React + Vite`，生产态构建产物为：

```text
/opt/genergi/current/apps/web/dist
```

由 `nginx` 直接作为静态站点根目录。

### 4. `apps/api` 与 `apps/worker` 走 systemd

两个长任务服务必须由 `systemd` 守护：

- `genergi-api.service`
- `genergi-worker.service`

并统一读取：

```text
/opt/genergi/shared.env
```

## nginx 责任边界

`nginx` 只负责两件事：

1. 静态站点直出
2. `/api/` 反向代理到 `127.0.0.1:8787`

不再通过 `nginx -> docker web container -> app` 这类多余链路增加故障面。

## 健康检查

部署后最少要验证：

```bash
curl -fsS http://127.0.0.1:8787/api/health
curl -fsS -H 'Host: ai.genergius.com' http://127.0.0.1/api/health
systemctl is-active genergi-worker
```

## DNS 验收规则

`genergius.com` 可访问，不代表 `ai.genergius.com` 已经可访问。  
正式验收必须单独验证：

- `ai.genergius.com` 的公共 DNS 解析
- `ai.genergius.com` 的公网访问结果

不能只看 DNS 提供商后台截图。
